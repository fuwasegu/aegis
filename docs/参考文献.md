# 参考文献とインスパイア元 (References)

Aegisのアーキテクチャ設計およびコンセプトは、以下の文献や既存ツールから強いインスピレーションを受け、それらを統合・発展させる形で構築されています。

## 1. Process Context Engine 2.0 (PCE2.0)
* **URL:** https://zenn.dev/caphtech/articles/process-context-engine-2
* **何をどう参考にしたか:**
  * **「Actor-local Compiled Context」の概念:** RAGのように「人間の脳（検索と連想）」を模倣するのではなく、エージェントの「責任（タスク）」に合わせて事前に文脈をコンパイル（構築）して渡すという、Aegisのコアパラダイムの基礎。
  * **記憶の3層分離:** 記憶を「Execution Plane（現場）」「Control Plane（制御）」「Canonical Knowledge（正典）」に明確に切り分けるアーキテクチャ設計の土台。Aegisはここでいう「Control Plane」として機能する。
  * **「Context」から「Process」へ:** 単に記憶を渡すのではなく、記憶を使ってエージェントの「責任の遷移（プロセス）」をどう成立させるかという視点。

## 2. PageIndex (推論ベースのツリー探索)
* **URL:** https://zenn.dev/knowledgesense/articles/148dfe2ca1d146
  * *(参考動画: https://www.youtube.com/watch?v=MMvQr3Sv4Yo)*
* **何をどう参考にしたか:**
  * **ミクロなグラフトラバーサルの手法:** テキストを文字数でチャンク分割するのではなく、Markdownの見出し（AST）などの木構造を維持したまま、LLMに目次を渡して必要な章（ノード）を推論・抽出させるアプローチ。Aegisが特定の規約ファイルから必要なセクションだけを切り出す際の検索アルゴリズムとしてエッセンスを取り入れている。

## 3. claude-mem (Data Plane記憶のデファクト)
* **URL:** https://github.com/thedotmack/claude-mem
* **何をどう参考にしたか:**
  * **アンチパターンの明確化（差別化）:** 現在主流の「会話ログを要約してベクトルDBに突っ込み、RAGで引き出す」というボトムアップ（履歴型）アプローチの事例として調査。
  * **Aegisの立ち位置の決定:** claude-memが「過去の作業履歴」を思い出すData Planeのツールであるのに対し、Aegisは「未来の設計規約」をトップダウンで強制するControl Planeのツール（アーキテクト）であるという、明確な役割分担と勝機を見出すきっかけとなった。

## 4. MAGS (Memory And Guidance System)
* **URL:** https://github.com/doancan/mags
* **何をどう参考にしたか:**
  * **プロセスの強制（Inversion of Control）のUX:** インストール時にプロジェクトのスタックを検知し、エージェントのシステムプロンプト（`CLAUDE.md`）を書き換えてルールを強制する、という完璧なオンボーディング体験を丸ごとインスパイア（吸収）。
  * **Hookによるサイレント・ロード:** セッション開始時に、検索するまでもない大前提のプロジェクト憲法を裏側で自動ロードする仕組み。
  * *※Aegisは、MAGSの優れたプロセス制御UXを継承しつつ、裏側の検索エンジンを「TF-IDF/Embedding」から「Plan駆動のDAGコンテキスト・コンパイル」に置き換えることで、より厳格なアーキテクチャ統制を実現する。*
