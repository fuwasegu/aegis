# Aegis - プロジェクト計画 v2

> **ステータス:** Draft for Review
> **前版:** [プロジェクト計画v1.md](./プロジェクト計画v1.md)
> **更新日:** 2026-03-12

本文書は v1 で提示した概念設計を、実装可能な水準まで具体化したものである。
議論を通じて確定した設計判断を **Normative**（規範）として記載し、
未確定の事項は **Deferred Decisions**（保留判断）として明示的に分離する。

---

## 1. 問題設定と設計原則

### 1.1 解決する課題

現在のコーディング AI エージェントは、プロジェクトの文脈把握を RAG（検索拡張生成）に依存している。
しかし RAG には**「意味の断絶（セマンティック・ギャップ）」**がある。

> 「ブックマークの並び替え機能を作って」と指示した際、エージェントは「ブックマーク」で検索する。
> しかし本当に必要な知識は「UseCase はオーケストレーションに徹する」
> 「DDD におけるファーストクラスコレクションの運用」といった抽象的なアーキテクチャ規約である。

結果として、適切な設計規約がヒットせず、AI はトランザクションスクリプト的な低品質コードを生成する。
人間が都度レビューしドキュメントを明示的に読ませる必要があり、開発体験を損なっている。

### 1.2 Aegis のミッション

AI の記憶システムを「人間の連想（検索）」から**「ソフトウェアの依存関係解決（コンパイル）」**へ
パラダイムシフトさせ、プロジェクトの設計規約を強制（Enforce）する次世代の MCP サーバーを構築する。

### 1.3 設計原則

| ID | 原則 | 根拠 |
|----|------|------|
| P-1 | **決定的であること** | 同一入力に対して同一のコンテキストを返す。LLM 推論に依存しない |
| P-2 | **信頼境界を明示すること** | 決定的に解決された知識と推論由来の知識を混ぜない |
| P-3 | **人間が承認した知識のみを正典とすること** | DAG の汚染を防ぐ |
| P-4 | **監査可能であること** | どの知識版でコードが生成されたかを追跡できる |
| P-5 | **エージェント非依存であること** | Control Plane は特定の AI エージェントに依存しない |

## 2. Aegis の 3 層モデル

v1 の PCE2.0 概念を踏襲しつつ、各層の責務を厳密に再定義する。

```
┌─ Execution Plane ─────────────────────────────────┐
│ エージェント固有。Plan 作成、コード生成、レビュー。    │
│ Aegis は関与しない。アダプタ層のみ提供。              │
└───────────────────────────────────────────────────┘
         │ compile_context()          ▲ observe()
         ▼                            │
┌─ Control Plane (Aegis Core) ──────────────────────┐
│ Read: 決定的グラフ探索 + 推論拡張                     │
│ Write: Observation → Proposed → Canonical           │
│ Audit: Snapshot による知識版追跡                      │
└───────────────────────────────────────────────────┘
         │
         ▼
┌─ Canonical Knowledge ─────────────────────────────┐
│ 人間が承認した構造化ドキュメント群 + DAG              │
│ SQLite に保存。バージョン管理あり。                    │
└───────────────────────────────────────────────────┘
```

**Normative:**
- Execution Plane と Control Plane の境界は MCP プロトコルで定義する
- Control Plane はエージェント固有の知識を一切持たない（P-5）
- Canonical Knowledge の変更は必ず Proposed Layer を経由する（P-3）

## 3. Read フェーズ

### 3.1 API

```
aegis.compile_context(request) -> CompiledContext
```

#### Request

```typescript
interface CompileRequest {
  // 一次キー（決定的ルーティング）
  target_files: string[]       // 編集対象ファイルパス（必須）
  target_layers?: string[]     // 明示レイヤ指定（省略時はパスから推論）
  command?: string             // scaffold, refactor, review 等

  // 二次キー（推論による拡張探索）
  plan?: string                // エージェントの実行 Plan（自然言語）
}
```

#### Response

```typescript
interface CompiledContext {
  base: {
    documents: ResolvedDoc[]
    resolution_path: Edge[]    // どのエッジを辿ったかの透明性ログ
    templates?: CodeSkeleton[]
  }

  expanded?: {
    documents: ResolvedDoc[]
    confidence: number
    reasoning: string
    resolution_path: Edge[]
  }

  compile_id: string             // この呼び出しの一意識別子
  snapshot_id: string              // 知識版の識別子（content-addressable hash）
  knowledge_version: number        // Canonical Knowledge のバージョン（単調増加）
}
```

**Normative:**
- `target_files` のみで最低限のコンテキストが得られる。`plan` がなくても動作する
- `base` は決定的ルーティングのみで構成される。SLM は関与しない（P-1, P-2）
- `expanded` は `base` と構造的に分離される。SLM が停止しても `base` に影響しない
- 戻り値には必ず `compile_id`（呼び出し単位）と `snapshot_id`（知識版単位）を含める（P-4）
- `resolution_path` により、なぜそのドキュメントが選ばれたかを検証可能にする
- `path_requires` で複数のパターンが一致した場合は**全て採用**する（union semantics）。
  `specificity` と `priority` はドキュメントの表示順序にのみ影響し、
  conflict resolution（排他選択）には使用しない

### 3.2 決定的ルーティング

`base` の解決は以下の 4 ステップで行う。全ステップで LLM/SLM は使用しない。

```
Step 1: path_requires
  target_files の各パスに対して、edges テーブルの glob パターンをマッチング。
  specificity DESC → priority ASC の順で評価。
  複数パターンが一致した場合は全て採用する（union semantics）。
  ordering は返却時のドキュメント表示順にのみ影響する。

Step 2: layer_requires
  target_files のパスからレイヤ名を layer_rules テーブル（Canonical 管理）で解決し、
  layer_requires エッジを引く。

Step 3: command_requires
  command が指定されていれば、対応するエッジを引く。

Step 4: doc_depends_on の推移的閉包
  Step 1-3 で集めた doc_id を起点に、doc_depends_on エッジを
  再帰的に辿り、依存先ドキュメントを全て収集する。
  最大深度 10 の安全弁を設ける。
```

### 3.3 推論拡張（Expanded Context）

`plan` が指定された場合にのみ実行する。

```
1. SLM が plan を解析し、Intent Tag を抽出
   （例: "collection_operation", "state_mutation"）
2. tag_mappings テーブルからタグに紐づくドキュメントを取得
3. base と重複するドキュメントを除外
4. confidence と reasoning を付与して返却
```

**Normative:**
- Intent Tag → ドキュメントのマッピングは `tag_mappings` テーブルに保存し、
  Canonical DAG（`edges` テーブル）には混ぜない（P-2）
- `expanded` が空でもエラーにしない。あくまで補助的な拡張である

### 3.4 エッジの型定義

```typescript
type EdgeType =
  | "path_requires"      // ファイルパスパターン → ドキュメント
  | "layer_requires"     // アーキテクチャレイヤ → ドキュメント
  | "command_requires"   // コマンド → ドキュメント
  | "doc_depends_on"     // ドキュメント → ドキュメント（DAG 制約あり）

interface Edge {
  edge_id: string
  source_type: "path" | "layer" | "command" | "doc"
  source_value: string   // glob pattern / layer name / command name / doc_id
  target_doc_id: string
  edge_type: EdgeType
  priority: number       // 低い値 = 高い優先度
  specificity: number    // glob のセグメント数（具体性）
  status: "draft" | "proposed" | "approved" | "deprecated"
}
```

## 4. Write フェーズ

### 4.1 3 層モデル

```
┌──────────────────────────────────────────────────────┐
│  Observation Layer（自動・無制限）                       │
│  事実の記録。構造化は不要。                               │
│  retention: 90 日で自動アーカイブ。                       │
├──────────────────────────────────────────────────────┤
│  Proposed Layer（SLM 抽出・人間未承認）                   │
│  SLM が Observation から構造化した仮説。                  │
│  ステータス: pending / approved / rejected / withdrawn    │
├──────────────────────────────────────────────────────┤
│  Canonical Layer（人間承認済み・DAG に反映）               │
│  Read フェーズで使用される唯一の知識源。                    │
│  全変更は Proposed 経由。直接編集不可。                     │
└──────────────────────────────────────────────────────┘
```

### 4.2 ライフサイクル

```
[イベント発生]
  │  PR マージ / レビュー修正 / コンパイルミス検知 / 手動ノート
  ▼
[Observation Layer に自動記録]
  │  aegis.observe(event)
  ▼
[バッチまたはトリガーで SLM が分析]
  │  新規エッジ提案 / ドキュメント更新案 / 新規ドキュメント案
  ▼
[Proposed Layer に配置]
  │  aegis.propose(proposal)
  │  根拠となる Observation へのリンク付き
  ▼
[人間がレビュー]
  │  approve → Canonical Layer にマージ + DAG 更新
  │  reject  → 理由を記録
  │  modify  → 修正して approve
  ▼
[Canonical Layer 更新]
  │  knowledge_version をインクリメント
  │  Snapshot を発行
  └→ 以降の compile_context() に反映
```

### 4.3 Write API

```typescript
// Observation（自動、エージェントが呼ぶ）
// Discriminated union: event_type ごとに必須フィールドが異なる
aegis.observe(event:
  | {
      event_type: "compile_miss"
      related_compile_id: string    // 必須: どの呼び出しで起きたか
      related_snapshot_id: string   // 必須: どの知識版で起きたか
      payload: { target_files: string[], missing_doc?: string, review_comment: string }
    }
  | {
      event_type: "review_correction"
      related_snapshot_id?: string  // 任意: 紐づく知識版があれば
      payload: { file_path: string, correction: string }
    }
  | {
      event_type: "pr_merged"
      payload: { pr_id: string, summary: string, files_changed: string[] }
    }
  | {
      event_type: "manual_note"
      payload: { content: string }
    }
): ObservationId

// Propose（SLM バッチ or 手動トリガー）
aegis.propose(proposal: {
  proposal_type: "add_edge" | "update_doc" | "new_doc" | "deprecate" | "bootstrap"
  payload: object
  evidence: ObservationId[]
}): ProposalId

// Approve（人間のみ）
aegis.approve(proposal_id: string, modifications?: object): CanonicalVersion
aegis.reject(proposal_id: string, reason: string): void
```

### 4.4 コンパイルミスのフィードバックループ

```
1. aegis.compile_context() → compile_id: "cmp_001", snapshot_id: "abc123"
2. エージェントがコード生成
3. 人間レビューで規約違反を発見
4. aegis.observe({
     event_type: "compile_miss",
     related_compile_id: "cmp_001",
     related_snapshot_id: "abc123",
     payload: { target_files, missing_doc, review_comment }
   })
5. SLM が分析 → aegis.propose({ type: "add_edge", evidence: [...] })
6. 人間が approve → DAG にエッジ追加
7. 次回から同一パターンで自動解決
```

**Normative:**
- Canonical への昇格は必ず人間の承認を経由する（P-3）
- Observation は自動記録。retention 超過分は自動アーカイブ
- Proposal は必ず 1 つ以上の Observation を根拠として持つ
- `observe` は discriminated union とし、`compile_miss` では `related_compile_id` と `related_snapshot_id` を必須とする

## 5. Canonical Knowledge のデータモデル

### 5.1 技術選定

**SQLite を採用する。** グラフ DB は採用しない。

理由:
- Aegis の主要操作は高頻度な任意グラフ分析ではなく、
  パス/レイヤ/コマンドからの決定的探索と `doc_depends_on` の閉包取得である
- 再帰 CTE とトランザクションで十分に実装可能
- 可搬性、デバッグ性、ローカル導入の軽さで優位

### 5.2 論理スキーマ

#### Canonical Knowledge Layer

```sql
CREATE TABLE documents (
    doc_id          TEXT PRIMARY KEY,
    title           TEXT NOT NULL,
    kind            TEXT NOT NULL
                    CHECK (kind IN ('guideline', 'pattern', 'constraint', 'template', 'reference')),
    content         TEXT NOT NULL,
    content_hash    TEXT NOT NULL,
    status          TEXT NOT NULL DEFAULT 'draft'
                    CHECK (status IN ('draft', 'proposed', 'approved', 'deprecated')),
    created_at      TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ', 'now')),
    updated_at      TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ', 'now'))
);

CREATE TABLE edges (
    edge_id         TEXT PRIMARY KEY,
    source_type     TEXT NOT NULL
                    CHECK (source_type IN ('path', 'layer', 'command', 'doc')),
    source_value    TEXT NOT NULL,
    target_doc_id   TEXT NOT NULL REFERENCES documents(doc_id),
    edge_type       TEXT NOT NULL
                    CHECK (edge_type IN ('path_requires', 'layer_requires',
                                         'command_requires', 'doc_depends_on')),
    priority        INTEGER NOT NULL DEFAULT 100,
    specificity     INTEGER NOT NULL DEFAULT 0,
    status          TEXT NOT NULL DEFAULT 'draft'
                    CHECK (status IN ('draft', 'proposed', 'approved', 'deprecated')),
    created_at      TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ', 'now'))
);

-- パス → レイヤ の決定的マッピングルール（Canonical 管理）
-- compile_context() の Step 2 で使用
CREATE TABLE layer_rules (
    rule_id         TEXT PRIMARY KEY,
    path_pattern    TEXT NOT NULL,       -- glob pattern (e.g. "app/UseCases/**")
    layer_name      TEXT NOT NULL,       -- 解決先レイヤ名 (e.g. "UseCase")
    priority        INTEGER NOT NULL DEFAULT 100,
    specificity     INTEGER NOT NULL DEFAULT 0,
    status          TEXT NOT NULL DEFAULT 'draft'
                    CHECK (status IN ('draft', 'proposed', 'approved', 'deprecated')),
    created_at      TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ', 'now'))
);
```

#### Observation Layer

#### Snapshot / Audit Layer

```sql
    rule_id         TEXT NOT NULL,
    path_pattern    TEXT NOT NULL,

    priority        INTEGER NOT NULL,
    specificity     INTEGER NOT NULL,
    PRIMARY KEY (snapshot_id, rule_id)
);

CREATE TABLE knowledge_meta (
    id              INTEGER PRIMARY KEY CHECK (id = 1),
    current_version INTEGER NOT NULL DEFAULT 0,
    last_updated_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ', 'now'))
);

-- init の実行記録（provenance）
CREATE TABLE init_manifest (
    id                  INTEGER PRIMARY KEY CHECK (id = 1),
    template_id         TEXT NOT NULL,
    template_version    TEXT NOT NULL,
    preview_hash        TEXT NOT NULL,

    seed_counts         TEXT NOT NULL,          -- JSON: { documents, edges, layer_rules }
    created_at          TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ', 'now'))

```

## 6. Snapshot/Audit と不変条件

### 6.1 不変条件（Invariants）

```
INV-1: Canonical の一貫性
  compile_context() が参照するのは status='approved' の documents/edges のみ。
  proposed/draft/deprecated は Read パスに入らない。

INV-2: DAG 制約
  edge_type='doc_depends_on' かつ status='approved' のエッジ群は
  有向非巡回グラフ（DAG）を構成する。
  approve 操作時にサイクル検知を実行し、検知した場合は approve を拒否する。

INV-3: Snapshot の不変性
  snapshot_docs, snapshot_edges は INSERT ONLY。
  一度作成されたスナップショットは変更・削除不可。

INV-4: バージョン単調増加
  knowledge_meta.current_version は approve/deprecate 操作ごとに +1。
  決して減少しない。構造変更（エッジ追加）と内容変更（ドキュメント更新）の
  両方でインクリメントされる。

INV-5: 監査追跡の完全性
  compile_context() の戻り値には必ず compile_id と snapshot_id を含む。
  compile_id は呼び出し単位の一意識別子であり、同一 snapshot_id に対する
  複数回の呼び出しを区別できる。
  observe(compile_miss) は必ず related_compile_id と related_snapshot_id を持つ。
  compile_id により「どの呼び出しで何が返されたか」を完全に追跡可能にする。

INV-6: 権限分離
  Agent Surface のツールは Canonical Knowledge を直接変更する能力を持たない。
  Canonical を変更できるのは Admin Surface の approve_proposal と init_confirm のみ。
  propose は MCP に公開しない。
```

### 6.2 Snapshot 発行ルール

- Snapshot は DAG に変更（approve/deprecate）があったときにのみ作成する
- `compile_context()` のたびには作成しない
- `knowledge_meta.current_version` が前回 Snapshot と同一なら既存 Snapshot を返す
- Snapshot ID は `approved` な `documents`, `edges`, `layer_rules` の正規化シリアライズの SHA-256
- コピー方式を採用する（参照方式は後からの edge 削除で整合性が壊れるため）

### 6.3 approve 操作のトランザクション要件

```
1. proposal.status == 'pending' を確認
2. proposal_type に応じて documents/edges を更新
3. doc_depends_on の場合、サイクル検知を実行（再帰 CTE）
4. proposal.status を 'approved' に更新
5. knowledge_meta.current_version をインクリメント
6. 新しい Snapshot を作成
7. 上記 1-6 を単一トランザクションで実行
```

### 6.4 サイクル検知

```sql
WITH RECURSIVE reachable(doc_id) AS (
    SELECT ?                        -- target_doc_id（追加先）
    UNION
    SELECT e.target_doc_id
    FROM reachable r
    JOIN edges e ON e.source_type = 'doc'
                AND e.source_value = r.doc_id
                AND e.edge_type = 'doc_depends_on'
                AND e.status = 'approved'
)
SELECT EXISTS (
    SELECT 1 FROM reachable WHERE doc_id = ?  -- source_doc_id
);
-- true ならサイクルが発生するため approve を拒否
```

## 7. 境界設計

### 7.1 Deterministic Core（信頼境界の内側）

以下の要素は決定的であり、LLM/SLM に依存しない。

| 要素 | 保存先 | 変更方法 |
|------|--------|----------|
| `documents` | SQLite `documents` テーブル | Proposed → approve のみ |
| `edges` (path/layer/command/doc) | SQLite `edges` テーブル | Proposed → approve のみ |
| `layer_rules` (path→layer) | SQLite `layer_rules` テーブル | Proposed → approve のみ |
| グラフ探索アルゴリズム | アプリケーションコード | コード変更のみ |

### 7.2 Inferred Expansion（信頼境界の外側）

以下の要素は SLM の推論に依存し、非決定的である。

| 要素 | 保存先 | 特性 |
|------|--------|------|
| Intent Tag 抽出 | 一時的（メモリ上） | plan ごとに再計算 |
| `tag_mappings` | SQLite `tag_mappings` テーブル | Canonical DAG とは別管理 |
| Observation → Proposal の変換 | SLM バッチ | 人間承認前の仮説 |

### 7.3 エージェントアダプタ（Execution Plane 境界）

**Normative:**
- Control Plane の API はエージェント非依存（P-5）
- エージェント固有の統合（`CLAUDE.md` ハック等）はアダプタ層に隔離する
- アダプタは Aegis Core の概念モデルに含めない

```
aegis/
├── core/           # Control Plane + Canonical Knowledge
│   ├── read/       # compile_context 実装
│   ├── write/      # observe / propose / approve 実装
│   ├── audit/      # snapshot / 監査
│   └── store/      # SQLite スキーマ・リポジトリ
├── expansion/      # Inferred Expansion（SLM 連携）
│   ├── intent/     # Intent Tag 抽出
│   └── analyzer/   # Observation → Proposal 変換
└── adapters/       # エージェント固有統合
    └── claude/     # CLAUDE.md 生成、Hook 設定等
```

## 8. Init（ブートストラップ）

### 8.1 設計方針

`init` は「賢く自動生成する機能」ではなく、
**「決定的に初期グラフを敷設するブートストラップ」**である。

- LLM/SLM は使用しない（P-1）
- Canonical への変更は Proposed Layer を経由する（P-3）
- 初期投入は高信頼の最小集合に絞る。推論由来のデータは入れない

### 8.2 4 段階フロー

```
detect → preview → approve → materialize
```

#### Stage 1: detect

ファイルシステムの静的走査のみ。LLM は使用しない。

検出対象:
- マニフェストファイル（`composer.json`, `package.json`, `pyproject.toml`, `go.mod`）
- 依存パッケージ名
- ディレクトリ構造（`app/Domain`, `app/UseCases`, `src/Application` 等）
- 規約パスの存在

検出結果から `StackDetection` と `ProfileCandidate[]` を決定的に算出する。

#### Stage 2: preview

推定結果をそのまま適用せず、`InitPreview` として返す。
曖昧な場合は自動選択しない。トップ候補を提示して人間に選ばせる。

#### Stage 3: approve

`init` も Canonical 変更であるため、Proposed Layer を経由する。
実装上は `proposal_type: "bootstrap"` の proposal バッチを生成し、
人間の confirm で一括 approve する。

**Normative:**
- `init` は通常の `propose → approve` フローの特殊ケースとして実装する。
  proposal テーブルをバイパスしない（P-3）
- confirm 時に `preview_hash` の一致を検証し、TOCTOU を防止する

#### Stage 4: materialize

approve 完了後、以下を実行する:
- `documents`, `edges`, `layer_rules` が `status='approved'` で投入済み
- `knowledge_version = 1`
- 初回 Snapshot を発行
- DB 内の `init_manifest` テーブルに provenance を記録

### 8.3 Init API

```typescript
aegis.init_detect(project_root: string): InitPreview
aegis.init_confirm(preview_hash: string): CanonicalVersion
```

#### InitPreview

### 8.4 Template Manifest

テンプレートはファイル構造で管理する。

```
templates/
  laravel-ddd/
    manifest.yaml
    documents/
      ddd_fundamentals.md
      usecase_guidelines.md
      ...
    edges.yaml          # manifest.yaml 内で定義してもよい
    layer_rules.yaml    # manifest.yaml 内で定義してもよい
  generic-layered/
    manifest.yaml
    documents/
    ...
```

#### manifest.yaml スキーマ

```yaml
template_id: string           # 必須。一意識別子
version: string               # 必須。セマンティックバージョン
display_name: string
description: string

# ======== detect_signals ========
detect_signals:
  required:                    # 全て満たさないと候補にならない
    - type: "file_exists" | "dir_exists" | "package_dependency" | "dir_structure"
      path?: string            # file_exists, dir_exists 用
      file?: string            # package_dependency 用
      key?: string             # package_dependency 用（"require", "dependencies" 等）
      pattern?: string         # package_dependency, dir_structure 用

  boosters:                    # 満たすと confidence が上がる
    - type: ...                # required と同じ型
      weight: number           # 加重（高いほど confidence 上昇に寄与）

  confidence_thresholds:
    high: number               # boosters の合計 weight がこれ以上 → high
    medium: number             # これ以上 → medium。未満 → low

# ======== placeholders ========
placeholders:
  <name>:
    description: string
    required: boolean          # true なら解決できない場合 block warning
    detect_strategy: "first_match" | "composer_autoload" | "package_json_field"
    candidates?: string[]      # first_match 用
    ambiguity_policy: "first" | "block"   # 複数候補が存在する場合の挙動
    default: string | null

# ======== seed_documents ========
seed_documents:
  - doc_id: string
    title: string
    kind: string
    file: string               # documents/ 以下の相対パス

# ======== seed_edges ========
seed_edges:
  - source_type: string
    source_value: string       # {{placeholder}} 使用可
    target_doc_id: string
    edge_type: string
    priority: number
    # specificity は書かない（自動算出）

# ======== seed_layer_rules ========
seed_layer_rules:
  - path_pattern: string       # {{placeholder}} 使用可
    layer_name: string
    priority: number
    # specificity は書かない（自動算出）
    when:                      # 条件付き生成（構造化条件）
      placeholder: string
      operator: "is_not_null" | "is_null" | "equals"
      value?: string           # equals 用
```

### 8.5 specificity 自動算出

`specificity` は manifest に記載せず、`source_value` / `path_pattern` から
materialize 時に自動算出する。

**Normative:**
- `specificity` は派生値であり、手動指定を許可しない
- 算出ロジックの厳密定義は D-3 に委ねるが、glob のセグメント数をベースとする

### 8.6 初期スコープ

初期実装では以下の 2 テンプレートのみ対応する。

| template_id | 対象 |
|-------------|------|
| `laravel-ddd` | Laravel + DDD / Clean Architecture |
| `generic-layered` | レイヤードアーキテクチャ（言語非依存） |

**Normative:**
- `detect` と `preview` の精度をこの 2 系統で固めてから拡張する
- テンプレート追加は manifest.yaml のファイル追加のみで行える設計とする

### 8.7 初期投入の範囲

`init` で投入するのは高信頼の最小集合のみ。

**含めるもの:**
- ルート規約ドキュメント
- レイヤごとの基本ガイド
- 高信頼な `path_requires` エッジ
- 必須の `doc_depends_on` エッジ
- 最低限の `layer_rules`

**含めないもの:**
- 細かい feature 単位の path edge
- 推論由来の `tag_mappings`
- 曖昧なテンプレート群
- アダプタ固有の挙動（`CLAUDE.md` 等）

### 8.8 Re-init / Upgrade ポリシー

**Normative:**
- `init` は未初期化状態でのみ実行可能。初期化済みプロジェクトでは `init` を拒否する
- 初期化済みかどうかは `knowledge_meta.current_version >= 1` で判定する
- テンプレート更新の適用（upgrade）は本文書のスコープ外とし、
  将来的に別コマンドとして設計する

### 8.9 Provenance

`init` の実行記録は DB 内に保存する。ファイルのみの記録は監査の単一情報源にならない。

```sql
CREATE TABLE init_manifest (
    id                  INTEGER PRIMARY KEY CHECK (id = 1),
    template_id         TEXT NOT NULL,
    template_version    TEXT NOT NULL,
    preview_hash        TEXT NOT NULL,
    stack_detection     TEXT NOT NULL,          -- JSON: StackDetection
    selected_profile    TEXT NOT NULL,
    placeholders        TEXT NOT NULL,          -- JSON: 解決済み placeholder の map
    initial_snapshot_id TEXT NOT NULL REFERENCES snapshots(snapshot_id),
    seed_counts         TEXT NOT NULL,          -- JSON: { documents, edges, layer_rules }
    created_at          TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ', 'now'))
);
```

### 8.10 proposals テーブルの拡張

`init` を Proposed Layer 経由にするため、`proposal_type` に `bootstrap` を追加する。

**Normative:**
- `bootstrap` proposal は `init` でのみ生成される
- 1 つの `bootstrap` proposal が複数の documents/edges/layer_rules を含む（バッチ）
- approve 時の動作は通常の proposal と同一（サイクル検知、Snapshot 発行等）

```sql
-- proposals テーブルの CHECK 制約を拡張
CHECK (proposal_type IN ('add_edge', 'update_doc', 'new_doc', 'deprecate', 'bootstrap'))
```

## 9. MCP プロトコルマッピング

### 9.1 設計方針

Aegis の MCP 公開面は**権限境界**で 2 系統に分離する。

単一サーバーに `approve` まで載せると、エージェントが prompt injection 等で
Canonical を直接変更できる攻撃面を作る。これは P-3（人間承認のみを正典）に反する。

### 9.2 Surface 分離

```
┌─ Agent Surface ──────────────────────────────────┐
│ エージェントが日常的に呼ぶ。                         │
│ Canonical を直接変更する操作を一切持たない。           │
├──────────────────────────────────────────────────┤
│  aegis/compile_context    Read                     │
│  aegis/observe            Write (Observation のみ) │
│  aegis/get_compile_audit  Audit (読み取り専用)      │
│  aegis/init_detect        Init (preview まで)      │
└──────────────────────────────────────────────────┘

┌─ Admin Surface ──────────────────────────────────┐
│ 人間または人間が操作する UI/CLI だけが呼ぶ。          │
│ Canonical を変更する唯一の経路。                     │
├──────────────────────────────────────────────────┤
│  aegis/list_proposals     Proposal 一覧            │
│  aegis/get_proposal       Proposal 詳細            │
│  aegis/approve_proposal   Canonical 変更           │
│  aegis/reject_proposal    Proposal 却下            │
│  aegis/init_confirm       Init 承認                │
└──────────────────────────────────────────────────┘

┌─ Internal Only ──────────────────────────────────┐
│ MCP に公開しない。Aegis 内部プロセスのみ。            │
├──────────────────────────────────────────────────┤
│  propose                  SLM バッチが生成          │
│  create_snapshot          approve 時に自動実行      │
│  update_tag_mappings      SLM が管理               │
│  observation_analysis     SLM バッチ処理            │
└──────────────────────────────────────────────────┘
```

**Normative:**
- `propose` は MCP に公開しない。エージェントが送るべきは「事実」（observe）であり「仮説」（propose）ではない
- Agent Surface のツールは Canonical を直接変更する能力を持たない
- Admin Surface のツールはエージェントに公開しない
- 2 つの Surface は同一プロセス内で実装してもよいが、権限チェックで分離する

|------|------|
| Canonical Knowledge | 人間が承認した構造化ドキュメント群と DAG |
| Compile Context | ファイルパス等を起点にDAGを探索し、必要なドキュメントを結合する操作 |
| Base Context | 決定的ルーティングのみで解決されたコンテキスト |
| Expanded Context | SLM の推論で追加されたコンテキスト（信頼境界の外側） |
| DAG | 有向非巡回グラフ。doc_depends_on エッジで構成される |
| Snapshot | ある時点の Canonical Knowledge の不変コピー |
| Observation | 自動記録されたイベントログ（非構造化可） |
| Proposal | SLM または人間が作成した Canonical への変更提案 |
| Specificity | glob パターンの具体性を示す数値。セグメント数で算出 |
| Resolution Path | compile_context が辿ったエッジの列。透明性の担保に使用 |
| Knowledge Version | Canonical Knowledge のバージョン番号。構造変更と内容変更の両方でインクリメントされる |
| Compile ID | compile_context() の呼び出し単位の一意識別子。同一 snapshot に対する複数呼び出しを区別する |
| Layer Rule | ファイルパスパターンからアーキテクチャレイヤ名への決定的マッピングルール。Canonical 管理 |
| Union Semantics | path_requires で複数パターンが一致した場合に全て採用する動作。排他選択ではない |
| Init | Canonical Knowledge の初期グラフを敷設するブートストラップ操作。detect → preview → approve → materialize の 4 段階 |
| Template Manifest | init テンプレートの宣言ファイル。detect_signals, placeholders, seed データを定義する |
| Preview Hash | InitPreview の全生成物の正規化ハッシュ。confirm 時の TOCTOU 防止に使用 |
| Bootstrap Proposal | init 専用の proposal_type。複数の documents/edges/layer_rules を一括で含むバッチ proposal |
| Agent Surface | エージェントに公開する MCP ツール群。Canonical を直接変更する能力を持たない |
| Admin Surface | 人間に公開する MCP ツール群。Canonical を変更できる唯一の経路 |
