# 議題: テンプレートシステムの将来設計

**起票日**: 2026-03-12
**ステータス**: 決定済み → [ADR-006](adr/006-template-system-future-design.md)

---

> **重要: この議題はゼロベースで議論すること。**
>
> テンプレートシステムは Aegis の init フロー（`detect` → `confirm`）で使われ、
> プロジェクトの技術スタックに応じた seed documents を投入する仕組みである。
> 基本設計をそのまま実装したが、将来の拡張性や運用面の検討は不十分。
> テンプレートという概念自体が正しいかも含めて、ゼロから考えること。

---

## 背景

### 現在の仕組み

- `templates/` ディレクトリに `manifest.yaml` + Markdown ファイル群を配置
- `manifest.yaml` がプロファイル定義（required/booster signals）とドキュメント一覧を持つ
- `detector.ts` がプロジェクトを走査し、最も適合するプロファイルをスコアリングで選択
- `template-loader.ts` が `{{placeholder}}` を解決し、`when` 条件で条件分岐
- 選択されたテンプレートの全ドキュメント + エッジを Proposal として投入
- `upgrade.ts` がテンプレートのバージョンアップを検出し、差分を Proposal 化

### サクッとやってしまったこと

- テンプレートの構造（manifest.yaml のスキーマ）を深く議論せずに決めた
- プロファイルのスコアリングアルゴリズム（required × 100 + booster × 10）を直感で決めた
- テンプレートは Aegis パッケージに同梱する前提で、外部テンプレートの仕組みがない
- `when` 条件の評価方法が限定的（`signal_name === true` の単純マッチ）
- テンプレートのバージョニングが `template_version` 文字列比較のみ

## 議論してほしいこと

### 1. テンプレートという概念は正しいか

- 「技術スタックに応じた初期ドキュメントセット」をテンプレートで提供するのは妥当か？
- 代替案: ユーザーが自分で初期ドキュメントを書く、コミュニティが共有する、AI が生成する
- テンプレートの粒度は適切か？（1プロファイル = 1技術スタック）
- 複数テンプレートの合成（例: React + GraphQL + Auth）は必要か？

### 2. テンプレートの配布と管理

現在テンプレートは npm パッケージに同梱されている:

- **同梱方式**: 更新が npm publish に依存。ユーザー独自のテンプレート追加不可
- **外部リポジトリ方式**: テンプレートを別パッケージ/リポジトリで管理
- **ローカルオーバーライド方式**: `~/.aegis/templates/` やプロジェクト内にカスタムテンプレートを置ける
- **レジストリ方式**: コミュニティテンプレートを検索・インストール

どの方式が適切か？ 組み合わせるべきか？

### 3. プロファイルスコアリングの妥当性

現在のスコアリング:

```
score = Σ(required signal match) × 100 + Σ(booster signal match) × 10
```

- この重み付けは妥当か？
- 同点の場合はブロック（警告）。これは正しい挙動か？
- Signal の検出方法（ファイル存在チェック）は十分か？
  - `package.json` の中身（dependencies）を見るべきでは？
  - `Dockerfile` の内容でランタイムを判別すべきでは？

### 4. テンプレートのバージョニングとアップグレード

- `template_version` の比較方法（文字列比較 vs semver）
- アップグレードで「追加」はできるが「削除」はできない。deprecated 対応は？
- ユーザーがカスタマイズしたドキュメントとテンプレートの更新がコンフリクトしたら？
- アップグレード提案の自動検出タイミングは？

### 5. テンプレートの内容品質

- テンプレートに含めるドキュメントの「正解」は誰が決める？
- 各技術スタックのベストプラクティスは変化する。メンテナンスは現実的か？
- テンプレートが押し付けがましくないか？（ユーザーのプロジェクトに合わない可能性）

### 6. `when` 条件と Placeholder の表現力

- 現在の `when` は signal 名の単純な真偽のみ。複合条件（AND/OR）は不要か？
- Placeholder の解決ルールは十分柔軟か？
- テンプレート内でのループや条件分岐（Jinja2 的な）は必要か？

## 仮実装の参考情報

- `src/core/init/detector.ts`: スタック検出、スコアリング
- `src/core/init/template-loader.ts`: マニフェスト読み込み、placeholder 解決
- `src/core/init/engine.ts`: init フロー全体
- `src/core/init/upgrade.ts`: テンプレートアップグレード検出
- `templates/`: テンプレートの実体

## 期待するアウトプット

1. テンプレート概念の妥当性評価
2. 配布・管理方式の決定
3. スコアリングアルゴリズムの改善案
4. バージョニング/アップグレード戦略
5. テンプレートのコミュニティ展開の是非
