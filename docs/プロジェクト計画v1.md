# Aegis - アーキテクチャ＆設計思想 検討記録

## 1. プロジェクトの背景と解決したい課題
現在のコーディングAIエージェント（Claude Code, Cursor等）は、プロジェクトの文脈を把握するためにRAG（検索拡張生成）やベクトル検索に依存している。しかし、このアプローチには**「意味の断絶（セマンティック・ギャップ）」**という致命的な課題がある。

* **課題の具体例:** 「ブックマークの並び替え機能を作って」と指示した際、エージェントは「ブックマーク」という単語で検索をかける。しかし、本当に必要な知識は「UseCaseはオーケストレーションに徹する」「DDDにおけるファーストクラスコレクションの運用」といった抽象的なアーキテクチャ規約である。
* **結果:** 適切な設計規約がヒットせず、AIはトランザクションスクリプト的な低品質なコードを生成してしまう。これを防ぐために人間が都度レビューし、ドキュメントを明示的に読ませる必要があり、開発体験を損なっている。

本プロジェクトの目的は、AIの記憶システムを「人間の連想（検索）」から**「ソフトウェアの依存関係解決（コンパイル）」**へとパラダイムシフトさせ、プロジェクトの設計規約を強制（Enforce）し、無秩序なコードからプロジェクトを守る絶対の盾となる次世代のMCPサーバー「Aegis」を構築することである。

## 2. コア・パラダイムシフト
本設計は、PCE2.0（Process Context Engine）の概念を取り入れ、記憶を以下の3層に分離して管理する。

1. **Execution Plane（現場の記憶）:** エージェントが持つ短期的なタスク実行用のコンテキスト。タスク完了後に破棄される。
2. **Control Plane（制御層の記憶）:** Aegis本体が担う。ドキュメント間の依存関係（DAG）を管理し、現場の責任（タスク）に合わせて必要な文脈を結合して渡すルーター（戦術情報システム）。
3. **Canonical Knowledge（正典知識）:** 構造化され、システムによって承認された公式ドキュメント群。

## 3. 試行錯誤とアイデアの変遷
ここに至るまでに、いくつかのアーキテクチャ案を検討・棄却・統合してきた。

* **✖ 棄却案: Qwenによる推論ルーター型**
  * *内容:* ドキュメント群をQwenに読ませ、「このタスクにはこのドキュメントが必要」とLLMに推論させる。
  * *理由:* 「ブックマークの並び替え」から「DDDのコレクション制約」をゼロショットで推論するのはLLMには困難であり、非決定的（ブラックボックス）になるため棄却。
* **△ 保留/部分採用案: CQRSとイベントソーシング型**
  * *内容:* 会話やPRのログを「イベント」として記録し、必要な時に読み取り用モデルを生成する。
  * *結果:* 「Read（抽出）」フェーズには不向きだが、「Write（記憶の定着）」フェーズにおける非同期インジェストの仕組みとして採用。
* **◎ 採用案: DAG（有向非巡回グラフ）とトリプル推論型**
  * *内容:* ドキュメントをノードとし、「AはBに依存する(requires)」といったエッジ（関係性）を持たせる。LLMの推論を挟まず、グラフ探索アルゴリズムで決定的にコンテキストを抽出する。

## 4. 既存ツールの事例調査と差別化
業界のデファクトスタンダードとなっている外部記憶ツールを調査し、Aegisの立ち位置と優位性を明確にした。

### 事例1: `claude-mem` (自動要約・ハイブリッドRAG型)
* **アプローチ:** セッションごとの会話やエラーログを裏側で自動要約し、SQLite/ChromaDBに保存してRAGで引き出す「ボトムアップ・履歴型」。
* **評価:** 「過去のバグ修正」などを思い出すのには優れるが、未来のコードに対してトップダウンでアーキテクチャ規約を強制する力はない。Data Planeの記憶ツール。

### 事例2: `MAGS` (Memory And Guidance System)
* **アプローチ:** Event Hookを利用してセッション開始時にプロジェクト憲法を自動ロードし、スタックを検知して `CLAUDE.md` を書き換える「トップダウン・規約型」。
* **評価:** プロセスを縛るUXやインストール体験は非常に優れている（インスパイア対象）。しかし、検索エンジン自体はTF-IDFやEmbeddingに依存しており、文脈の動的な結合（コンパイル）までは踏み込めていない。

### Aegis の勝機
MAGSの優れた「プロセスの強制（UX）」を吸収しつつ、心臓部の検索エンジンを「RAG」から**「Plan駆動のDAGコンテキスト・コンパイラ」**に置き換えることで、世界初の「完全に設計規約を遵守させるアーキテクトAI」が実現できる。

## 5. 最終的なアーキテクチャ設計

### A. プロセスの強制（Inversion of Control）
* **ゼロコンフィグInit:** `init` コマンドでプロジェクトの技術スタックを検知し、初期ノード（ルートドキュメント）を配置。
* **`CLAUDE.md` のハック:** インストール時にエージェントのシステムプロンプトを書き換え、**「コードを書く前に必ずPlanを立て、AegisにContextを要求し、規約違反がないか自己レビューせよ」**というプロセスを強制する絶対の盾（ルール）を展開する。

### B. Readフェーズ（コンテキストの抽出とコンパイル）
1. **Plan駆動エントリー（フェーズドアレイレーダー）:** エージェントは「検索キーワード」ではなく、「実行予定のPlan」と「編集対象のファイルパス（例: `app/UseCases/`）」をMCPツールに渡す。
2. **Intent Tagging:** Qwen（SLM）がPlanを解析し、「状態更新」「コレクション操作」などの抽象タグを付与する。
3. **決定的なグラフ探索:** ファイルパスとタグを起点に、裏側のDAG（トリプル構造）をアルゴリズムで探索。
4. **コンパイル（情報処理・迎撃）:** 「UseCase層を触るなら `usecase_guidelines.md`」「コレクション操作なら `collection_objects.md`」をノイズゼロで結合し、エージェントに叩きつける。

### C. Writeフェーズ（記憶の昇格）
* 人間が直接Markdownを編集するのではなく、ツール経由でのみドキュメントのインジェストを許可する。
* Qwenが入力データ（PRの議論など）から依存関係（エッジ）を推論し、構造化された「Canonical Knowledge（正典）」として昇格させる。
