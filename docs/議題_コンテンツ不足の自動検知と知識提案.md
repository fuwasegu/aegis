# 議題: コンテンツ不足の自動検知と知識提案

**起票日**: 2026-03-25
**ステータス**: Phase 1 完了

---

## 背景

現在の Observation → Proposal パイプラインは、**エッジ（パスとドキュメントの紐付け）の不足**しか自動検知できない。

### 現状の自動分析

| Analyzer | トリガー | 生成する Proposal | 制約 |
|---|---|---|---|
| `RuleBasedAnalyzer` | `compile_miss` + `missing_doc` あり | `add_edge` | `missing_doc` が指定されていなければ skip |
| `ReviewCorrectionAnalyzer` | `review_correction` + `target_doc_id` + `proposed_content` | `update_doc` | 両フィールド必須。なければ skip |
| `PrMergedAnalyzer` | `pr_merged` + カバーされていないパス | `add_edge` | ルートドキュメントへの紐付けのみ |
| `ManualNoteAnalyzer` | `manual_note` + hints | `update_doc` / `new_doc` | 人間が明示的にヒントを付ける必要あり |

### 対処できないケース

1. **ドキュメントの内容が不十分**: エッジは正しく張られているが、返されたドキュメントの内容がガイドラインとして不足している（例: エラーハンドリング規約はあるが、バリデーション規約が含まれていない）
2. **ドキュメント自体が存在しない**: そもそもその領域のガイドラインドキュメントが知識ベースに存在しない（例: テスト戦略のドキュメントがない）
3. **`compile_miss` の `missing_doc` なし**: エージェントが「何か足りない」と報告するが、具体的にどの doc_id が不足しているかを特定できない場合。現状は skip される

これらはすべて `compile_miss` の `review_comment` には情報が含まれているが、現行の RuleBasedAnalyzer はルールベースで `missing_doc` フィールドの有無だけを見るため、自然言語の `review_comment` から意味を汲み取れない。

## 目的

`compile_miss` の `review_comment` やその他のシグナルから、**エッジ不足だけでなくドキュメントの更新（`update_doc`）や新規作成（`new_doc`）の Proposal を自動生成できるようにする。**

## 解決すべき課題

### 1. 不足の種類の判別

`compile_miss` の `review_comment` から、以下のどれに該当するかを判別する必要がある:

| 種類 | 意味 | 適切な Proposal |
|---|---|---|
| エッジ不足 | ドキュメントは存在するが、対象パスと紐付いていない | `add_edge`（現行で対応済み） |
| 内容不足 | ドキュメントは返されたが、内容にガイドラインが不足 | `update_doc` |
| ドキュメント欠如 | そもそもその領域のドキュメントが存在しない | `new_doc` |

ルールベースでの判別は困難。`review_comment` の自然言語を解釈する必要がある。

### 2. コンテンツの生成

`update_doc` や `new_doc` を Proposal として生成するには、**更新後のコンテンツ / 新規ドキュメントの内容**が必要。

現行の `ReviewCorrectionAnalyzer` と `ManualNoteAnalyzer` は、`proposed_content` として人間が内容を渡すことを前提としている。自動検知の場合、コンテンツをどう生成するか:

- **SLM で生成**: `review_comment` + 既存ドキュメント + プロジェクトコンテキストから SLM がドラフトを生成
- **テンプレートベース**: `review_comment` を埋め込んだ定型ドキュメントを生成（品質は低いが決定的）
- **人間へのエスカレーション**: Proposal に「内容は人間が記述してください」というプレースホルダーを入れ、承認時に modifications で上書きする

### 3. Proposal の品質保証

SLM やルールで自動生成されたコンテンツが Proposal になると、的外れな提案が増えるリスクがある:

- 人間が承認する前提（P-3）は変わらないが、ノイズが増えるとレビュー負荷が上がる
- 信頼度スコアを付与して、閾値以下は skip する戦略
- 同一領域の `compile_miss` が N 件蓄積してから Proposal を生成する（1 件では判断しない）

### 4. 既存 Analyzer との関係

新しい分析ロジックをどこに置くか:

| 方式 | 概要 |
|---|---|
| 既存の `RuleBasedAnalyzer` を拡張 | `missing_doc` なしの `compile_miss` も処理対象にする |
| 新しい `ContentAnalyzer` を追加 | ObservationAnalyzer インターフェースの新しい実装 |
| SLM-Powered Analyzer | SLM を使う専用 Analyzer。SLM 無効時は skip |

`ObservationAnalyzer` はポートとして定義済み（`analyzer.ts`）なので、新しい実装を追加するのが自然。SLM 依存のロジックは SLM 無効時にグレースフルにスキップできる設計にすべき。

## 想定する実装方針

### Phase 1: パターンマッチングによる `update_doc` 提案

SLM 不要。`compile_miss` の `review_comment` + `target_doc_id` の組み合わせで、既存ドキュメントの更新を提案する。

- `target_doc_id` が指定されている → そのドキュメントの内容が不足
- `review_comment` をそのまま Proposal の `review_comment` に記録
- **コンテンツは生成しない**: 人間が承認時に modifications で内容を追加する運用

現行の `RuleBasedAnalyzer` が `missing_doc` なし + `target_doc_id` ありのケースを skip しているのを拾う形。

### Phase 2: SLM によるコンテンツドラフト生成

SLM を使い、`review_comment` + 既存ドキュメント + compile_context の結果から、更新後のコンテンツのドラフトを生成。

- `update_doc` Proposal に `proposed_content` を含める
- `new_doc` Proposal にドラフトを含める
- SLM 無効時は Phase 1 にフォールバック

### Phase 3: 蓄積分析

同一パス・同一ドキュメントへの `compile_miss` が複数蓄積した場合に、それらを統合して 1 つの Proposal を生成する。

- 単発の `compile_miss` ではノイズの可能性がある
- 複数のエージェントや複数のセッションで同じ不足が報告されたら、信頼度が高い

## 決定が必要なこと

1. Phase 1 のスコープ（`target_doc_id` ありのケースだけか、`review_comment` のキーワード分析も含めるか）
2. SLM を使う場合のモデル要件と生成プロンプトの設計方針
3. Proposal にコンテンツなしで生成する場合の承認 UX（Admin surface でどう見せるか）
4. 蓄積分析のしきい値と集約ロジック
