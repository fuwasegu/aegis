# 議題: 既存ドキュメントの取り込み戦略

**起票日**: 2026-03-12
**ステータス**: 決定済み → [ADR-002](adr/002-document-import-observation-wrapper.md)

---

> **重要: この議題はゼロベースで議論すること。**
>
> 現在の仮実装（`aegis_import_doc` ツール、正規表現ベースのコンテンツ分析）は
> あくまでプロトタイプであり、方針決定のたたき台ですらない。
> 既存のコードや設計に引っ張られず、「既存プロジェクトのドキュメントを
> Aegis にどう取り込むべきか」をゼロから考えてほしい。
>
> 仮実装の存在は忘れてよい。最善の設計が決まったら、仮実装は捨てるか書き直す。

---

## 解決すべき問題

Aegis は現在、**テンプレートの seed documents** からしかドキュメントを取り込めない。
つまり、ユーザーが既に持っている設計ドキュメントや運用ルール、ADR などは
Aegis の知識グラフに存在せず、コンテキストコンパイルに一切反映されない。

「既存プロジェクトに Aegis を入れたらすぐ効く」という体験を作るには、
既存ドキュメントの取り込み手段が不可欠。

## 前提条件

- Aegis は MCP サーバーとして動作する。呼び出し元は AI エージェント（Claude, GPT, Codex 等）または人間
- ドキュメントは多言語（日本語・英語・混在）
- ドキュメントのフォーマットは統一されていない（YAML frontmatter あり/なし、Markdown/プレーンテキスト、構造化/雑メモ）
- Aegis の Canonical Knowledge に入れるには `doc_id`, `title`, `kind`, `content`, DAG エッジ（`requires` 等）が必要
- Proposal → Approve のガバナンスフローは維持する（INV-6）
- SLM（llama.cpp、Qwen 3.5 4B/9B）がローカルで利用可能（ただし無効化されている場合もある）

## 議論してほしいこと

### 1. 取り込みの起点は何か

- MCP ツール呼び出し（エージェントが1ファイルずつ指定）？
- ディレクトリスキャン（`docs/` を一括走査）？
- init フローの一部として自動検出？
- 上記の組み合わせ？

### 2. メタデータ（kind, tags, doc_id, title）の決定方法

ドキュメントの種別やタグをどうやって決めるか。選択肢は幅広い:

- ユーザー（人間）が全部手動で指定
- 呼び出し元の AI エージェント（Claude/GPT）がドキュメントを読んで判断
- Aegis 内蔵の SLM が判断
- ルールベース（正規表現、キーワード等）
- フロントマターで明示
- 上記の組み合わせ

精度、速度、決定論性、多言語対応、SLM 無効時の挙動、ユーザー負荷、
それぞれのトレードオフを考慮すること。

### 3. DAG エッジの構築

取り込んだドキュメントを DAG のどこに接続するか。
孤立ノードでは `compile_context` に出てこないので、エッジがなければ意味がない。

- 誰がエッジを決める？
- 既存の seed documents との関係はどう推定する？
- path_requires, layer_requires, doc_depends_on のどれを使う？

### 4. ワークフローの設計

1ステップで完結するのか、複数ステップに分けるのか。

- 分析 → プレビュー → 確認のフロー（init の detect → confirm パターン）？
- 1ショットで proposal 作成して、approve/reject で調整？
- バッチ取り込み（ディレクトリ内の全 .md を一括処理）は必要？

### 5. doc_id の生成規約

非 ASCII（日本語等）のドキュメントタイトルからどう doc_id を生成するか。
そもそも自動生成すべきか、必ず明示させるべきか。

### 6. 仮実装で見えた課題（参考情報）

以下は仮実装（正規表現ベースのコンテンツ分析）で明らかになった問題点。
参考にはしてよいが、解決策は仮実装に縛られないこと。

- 正規表現による kind 分類は英語にしか効かない。日本語パターンを追加し始めたが、言語ごとにメンテするのは現実的でない
- タグ推論（14カテゴリ × 多言語）の組み合わせ爆発
- ファイル名ベースの doc_id/title 推定は精度が低い
- `# 見出し` からのタイトル抽出は比較的うまくいく
- コンテンツ分析の confidence が low の場合の扱いが曖昧
- 1ショットで proposal を作ると、やり直しが面倒（reject して再 import）

## 期待するアウトプット

1. 取り込み戦略の方針決定（上記の各論点に対する結論）
2. ワークフローの設計（ユーザー/エージェントがどういう手順で取り込むか）
3. 実装計画（MCP ツールの IF 設計、内部アーキテクチャ）
