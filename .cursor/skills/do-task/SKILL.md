# タスクの実施

`docs/tasks/*.md` のタスクの実施を命じられたら，以下のワークフローに従いタスクを実行します．

## Step1 実施するタスクを選択する

`docs/tasks/*.md` のタスクの中で， closed になっていないタスクを選ぶ．
その際，Priority や依存タスクを判断基準にすること．

## Step2 ブランチを切る

1. main ブランチへ移動
2. git pull して最新化
3. タスク名からブランチ名を決め，ブランチを新規で作りチェックアウト

## Step3 実装

Step１で選び取ったタスク内容を実施する

## Step4 レビュー依頼（Codex CLI・ヘッドレス）

Step3 の差分に対して **CriticalReview** を Codex に依頼する．**`codex review` に依存しない**．主に **`codex exec`（非対話・ヘッドレス）** で，下記のプロンプト雛形に **タスクファイルパス** と **ベースブランチ** を埋めて渡す．

### Step4 の中心ループ（必ずこの順で繰り返す）

Step4 は **一回きりのレビューでは終わらない**．次のループとする．

1. **初回レビュー** — プロンプト雛形を埋め，`codex exec` で CriticalReview を依頼する（下記「前提」〜「プロンプト雛形」に従う）．
2. **指摘への対応** — Codex の出力を読み，**ブロッキングになりうる指摘**から順にコード・テストで修正する．必要なら WIP コミットする．
3. **再レビュー** — **同じタスクファイル・同じ観点・同じプロンプト雛形**で，再度 `codex exec` を実行する（差分は更新後の `main...HEAD` / 作業ツリー）．
4. **終了条件** — Codex の結論が **「ブロッキングなし」相当**（重大な指摘が残っていない）と判断できるまで **2〜3 を繰り返す**．

人間のオペレータが Codex を手で叩く場合も，エージェントが代行する場合も，**このループが Step4 の定義**である．

### 前提（重要）

- **カスタムプロンプト**と **`codex review --uncommitted`** は併用できない．観点付きレビューは **`codex exec`** で行う．
- 差分の取り回しを安定させるため，**少なくとも WIP コミット**してからレビューする（未コミットだけを見せたい場合は，プロンプト内で `git diff` / `git status` の取得を明示的に指示する）．

### 依頼時に必ず含めるコンテキスト

- 対象タスクファイル: `docs/tasks/<id>-*.md` の **リポジトリ相対パス**（本文の「受け入れ条件」をレビュー対象に含める）．
- 比較基準: 例 `main`（またはチームのデフォルトブランチ名）．

### レビュー観点（必須）

次を漏れなく扱うよう依頼すること．

1. 致命的なバグの有無  
2. 潜在的なバグの有無  
3. タスクの受け入れ条件・仕様を満たしているか  
4. コードの品質・パフォーマンス  
5. 設計の妥当性  
6. （該当すれば）テストが受け入れ条件を押さえているか  

### プロンプト雛形（`codex exec` に渡す）

エージェントに **ファイル読取・git 操作**が必要なら，Codex の sandbox / 権限設定に合わせて `-c` 等で足す．

```text
あなたはこのリポジトリでコードレビューを行う．対話は不要．最後に箇条書きで結論を出すこと．

## タスク（必ず読む）
- ファイル: <TASK_MD_PATH>
- このファイルの「受け入れ条件」をチェックリストとしてレビューに使うこと．

## 差分
- ベースブランチ: <BASE_BRANCH>（例: main）
- まず `git fetch` が必要なら実行し，`git diff <BASE_BRANCH>...HEAD` および必要なら `git diff`（未コミット）で変更範囲を把握すること．
- 変更ファイルの周辺コンテキストが足りなければ，該当ソースを読んでよい．

## 依頼（CriticalReview）
次の観点で，重要度の高い順に指摘すること．指摘ゼロなら「ブロッキングなし」と明示すること．

1. 致命的なバグ  
2. 潜在的なバグ  
3. 受け入れ条件・仕様との整合  
4. コード品質・パフォーマンス  
5. 設計の妥当性  
6. テストの十分性（該当する場合）  

各指摘には，ファイル名・根拠・改善の方向性を簡潔に付けること．
```

### 実行例

```bash
# 雛形をファイルに保存しておく場合
# cat docs/templates/codex-critical-review.prompt | sed ... | codex exec -
```

（プロジェクトでテンプレートの置き場所と `npm run` ラッパーを決めてもよい．）

### （参考）フィードバックループの一文要約

レビュー受領 → 修正 → **同じ手順で再 `codex exec`** → 問題なければ Step5 へ．

## PR を作る準備

コミットには署名が必要なので，必ず人間のオペレータが行う，
あなたが行うのはその準備まで．

1. 対象の差分を stage する
2. コミットメッセージを考える（コミットはしない）
3. PullRequest の Note を考える（PullRequest を Open しない）

考えたコミットメッセージと PR Note を提示し，コミット＆プッシュを依頼してください
