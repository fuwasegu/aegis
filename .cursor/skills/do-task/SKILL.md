# タスクの実施

`docs/tasks/*.md` のタスクの実施を命じられたら，以下のワークフローに従いタスクを実行します．

## Step1 実施するタスクを選択する

`docs/tasks/*.md` のタスクの中で， closed になっていないタスクを選ぶ．
その際，Priority や依存タスクを判断基準にすること．

## Step2 ブランチを切る

1. main ブランチへ移動
2. git pull して最新化
3. タスク名からブランチ名を決め，ブランチを新規で作りチェックアウト

## Step3 実装

Step１で選び取ったタスク内容を実施する

## Step4 レビュー依頼（CriticalReview）

Step3 の差分に対して **CriticalReview** を Codex に依頼する．**Step4 は blocking gate であり，レビュー結果を受領するまで Step5 以降へ進んではならない**．**`codex review` に依存しない**．

### 同期・停止ルール（最重要）

- **レビュー依頼を送っただけでは Step4 完了ではない**．Codex の最終出力を受領し，内容を読める状態になって初めて 1 回のレビュー完了とみなす．
- **エージェントが代行する場合の第一選択は統合ターミナルの同期実行**．`.cursor/skills/do-task/scripts/codex-critical-review.sh` を前景で実行し，プロセス終了まで待つこと．
- **エディタ拡張や別セッションなど，このエージェントが完了を観測できない経路で依頼した場合**，依頼直後に停止し，ユーザーへ「レビュー結果の貼り付け」または「完了した」旨の連絡を求めること．その確認前に Step5 へ進まない．
- **ユーザーが明示的にレビュー省略を指示した場合のみ** Step4 をスキップできる．

### 推奨順（エージェントが完了を観測しやすい順）

1. **統合ターミナルでスクリプト直接実行** — エージェント自身が同期的に待機できるため最優先．下記「ターミナルから実行する」．ログを `tee` でファイルに残すとより確実．
2. **Codex Extension（エディタ）** — チャットでプロンプトを渡すか，`.cursor/skills/do-task/templates/codex-critical-review.prompt` の内容をコピーし，`<TASK_MD_PATH>` / `<BASE_BRANCH>` を手で置換して貼る．人間が画面で追いやすい．ただし，エージェントは結果を観測できないなら停止して共有待ちに入ること．

CLI の `codex exec` をヒアドキュメントだけで渡すと，長時間実行で**出力が途切れる**ことがあるため，上記いずれかを優先すること．

### Step4 の中心ループ（必ずこの順で繰り返す）

Step4 は **一回きりのレビューでは終わらない**．次のループとする．

1. **初回レビュー** — 下記「プロンプト雛形」に **タスクファイルパス** と **ベースブランチ** を埋め，CriticalReview を依頼する．
   **依頼を出しただけで次へ進まず，レビューの完了出力が返るまで待つこと．**
2. **指摘への対応** — Codex の出力を読み，**ブロッキングになりうる指摘**から順にコード・テストで修正する．
3. **再レビュー** — **同じタスクファイル・同じ観点・同じプロンプト**で再度レビューを依頼する（差分は下記「差分の見方」に従い，更新後の作業ツリー全体を対象にする）．
4. **終了条件** — 結論が **「ブロッキングなし」相当**（重大な指摘が残っていない）と判断できるまで **2〜3 を繰り返す**．

人間のオペレータが手で叩む場合も，エージェントが代行する場合も，**このループが Step4 の定義**である．**依頼だけして先に進むのは Step4 未完了**とみなす．

### 前提（重要）

- **カスタムプロンプト**と **`codex review --uncommitted`** は併用できない．観点付きレビューは **`codex exec`**（または拡張に同等の指示）で行う．

### 差分の見方（レビューで取りこぼさない）

- ベースブランチ（例: `main`）に対する変更の**全体**をレビュー対象とする．**WIP コミットは不要**（オペレータが署名付きでコミットする PR 準備フローと両立させるため）．
- CriticalReview を依頼するときは，プロンプトまたは口頭指示で次を必須とする:
  - `git status`
  - `git diff <BASE_BRANCH>` — ベースの先端に対する**作業ツリー・インデックス込み**の差分（未コミットを含む）．
  - 補助として `git diff <BASE_BRANCH>...HEAD`（ブランチ上のコミット済み差分のみ）を併用してもよい．

### 依頼時に必ず含めるコンテキスト

- 対象タスクファイル: `docs/tasks/<id>-*.md` の **リポジトリ相対パス**（本文の「受け入れ条件」をレビュー対象に含める）．
- 比較基準: 例 `main`（またはチームのデフォルトブランチ名）．

### レビュー観点（必須）

次を漏れなく扱うよう依頼すること．

1. 致命的なバグの有無
2. 潜在的なバグの有無
3. タスクの受け入れ条件・仕様を満たしているか
4. コードの品質・パフォーマンス
5. 設計の妥当性
6. （該当すれば）テストが受け入れ条件を押さえているか

### プロンプト雛形（リポジトリ管理）

全文は **`.cursor/skills/do-task/templates/codex-critical-review.prompt`** に置く．`<TASK_MD_PATH>` と `<BASE_BRANCH>` を置換して Codex に渡す．

### ターミナルから実行する（リポジトリルートで）

```bash
# 例: タスク MD とベースブランチを指定
bash .cursor/skills/do-task/scripts/codex-critical-review.sh docs/tasks/014-01-example.md main
```

ログ保存の例:

```bash
bash .cursor/skills/do-task/scripts/codex-critical-review.sh docs/tasks/014-01-example.md main 2>&1 | tee /tmp/codex-review.log
```

エージェントに **ファイル読取・git 操作**が必要なら，Codex の sandbox / 権限設定に合わせて `.cursor/skills/do-task/scripts/codex-critical-review.sh` 内の `codex exec` に `-c` 等を足す．

### （参考）フィードバックループの一文要約

レビュー受領 → 修正 → **同じ手順で再レビュー** → 問題なければ Step5 へ．

## Step5 フォーマット

biome を使ってフォーマットチェックする．

```bash
npm run lint:fix
```

自動で修正できないエラーは，手動で修正する．

## Step5 PR を作る準備

コミットには署名が必要なので，必ず人間のオペレータが行う，
あなたが行うのはその準備まで．

1. 対象の差分を stage する
2. コミットメッセージを考える（コミットはしない）
3. PullRequest の Note を考える（PullRequest を Open しない）

考えたコミットメッセージと PR Note を提示し，コミット＆プッシュを依頼してください
